Foundation

What Is Algo Trading & Why Build One?

Before writing a single line of code, you need to understand what you are actually building and why it beats manual trading for a rule-based system like this one.

An algo trading system is a computer program that monitors the market, identifies trade setups, and places orders — all automatically, according to a fixed set of rules that you define once. The computer never gets scared, never gets greedy, never misses a signal because it was looking at Twitter, and never breaks a rule because it "felt like" the trade would work.

That last sentence is why algos beat most manual traders over the long run. The edge in the ObsidianPivot system is not the indicators themselves — EMAs and ADX are publicly known. The edge is flawless, consistent rule execution. Every single day, every single candle, without exception. Only a computer can guarantee that.

What the System Does in Plain English
8:30 AM — Pre-market intelligence
Downloads yesterday's price data from Firstock. Calculates pivot levels (PP, R1, R2, S1, S2) — the exact prices where institutional money concentrates. Scores each stock 0–4 on the morning filter. Sends a Telegram summary to your phone.
9:15 AM — Market opens, system watches
System enters "watch mode." No trades for the first 15 minutes. It's gathering intelligence about the opening candle, gap size, and initial volume patterns.
Every 15 minutes — The 6-condition check
On every new candle, for every stock in your watchlist, the system checks all 6 entry conditions simultaneously. ADX ≥ 20, EMAs oriented, EMA gap ≥ 0.15%, price near pivot, candle confirms, volume ≥ 1.5x. All six must pass — one failure and it waits for the next candle.
Signal found — Order placed
Entry order goes to Firstock as a market order. Stop-loss order is placed on Firstock's servers immediately after — so your position is protected even if your internet dies. Telegram alert fires to your phone.
Trade management — automatic exits
System monitors on every candle. T1 hit → SL moves to breakeven automatically. T2 hit → exits remaining position. ADX drops below 20 mid-trade → exits immediately. 3:15 PM → hard close of all positions, no exceptions.
3:30 PM — Journal saved, system shuts down
Every trade written to trades_journal_firstock.csv. Full debug log saved. Firstock session logged out cleanly. Everything is ready for your evening review.

The Critical Principle: Separation of Concerns

The most important design decision in this system is that broker code only lives in 3 sections: S0 (imports), S8 (OrderManager), S11 (DataFetcher), and S13 (LoginManager). Every other section — S3 through S7, S9, S10, S12, S14, S15 — is pure Python logic with zero broker code.

This means if Firstock changes their API, or if you want to switch to a different broker tomorrow, you only touch 3 sections and everything else runs exactly the same. This is the professional way to architect any trading system.

Security rule — always read from environment variables

Never write API keys, passwords, or TOTP secrets directly in your Python file. If you ever accidentally push to GitHub, those credentials are exposed permanently and attackers will use them within hours. Always use os.getenv("KEY_NAME").

Why build Logging immediately after Config

Logging answers the question "did my code do what I think it did?" Without it, you are flying completely blind. The system creates two outputs: console (INFO level — what you see in real time) and file (DEBUG level — captures every condition check, every skip reason, every filter result). Review the log file every evening — it is more informative than watching the terminal live.

PYTHON
class PivotCalculator: @staticmethod def calculate(high: float, low: float, close: float) -> dict: # Classic pivot point formulas — used by institutions worldwide pp = (high + low + close) / 3 # Central pivot return { "PP": round(pp, 2), "R1": round((2 * pp) - low, 2), # Resistance 1 "R2": round(pp + (high - low), 2), # Resistance 2 "S1": round((2 * pp) - high, 2), # Support 1 "S2": round(pp - (high - low), 2), # Support 2 } @staticmethod def nearest_pivot(price: float, pivots: dict): # Find the closest pivot level within 0.2% of current price # If price is further than 0.2% from all levels → no signal for name, level in sorted(pivots.items(), key=lambda x: abs(x[1] - price)): if abs(price - level) / price <= Config.PIVOT_ENTRY_TOLERANCE: return name, level # Returns e.g. ("S1", 1185.20) return None, None @staticmethod def target_levels(entry_pivot: str, direction: str, pivots: dict): # Map entry pivot to T1 and T2 targets # Long: S2→S1→PP→R1→R2 (entry near S levels, target PP then R1) # Short: R2→R1→PP→S1→S2 (entry near R levels, target PP then S1) seq = ["S2","S1","PP","R1","R2"] if direction == "short": seq = list(reversed(seq)) idx = seq.index(entry_pivot) return pivots.get(seq[idx+1]), pivots.get(seq[idx+2])
How to test PivotCalculator before writing more code

Open Python interactive mode: python3. Import your file and test: from nse_algo_firstock import PivotCalculator. Then: PivotCalculator.calculate(2850, 2780, 2810). Compare the output with any online pivot calculator. If they match, your math is correct. Do this sanity check before moving to Section 4.

# This is the entire entry logic — 6 conditions, all must pass: # Condition 1: ADX ≥ 20 (trend must exist) c1 = ind["adx"] >= 20.0 # Condition 2: EMAs oriented correctly for our direction c2 = ind["ema_dir"] == (1 if direction == "long" else -1) # Condition 2b: EMAs not tangled (≤2 crossovers in 6 candles) c2b = ind["ema_crosses_6"] <= 2 # Condition 3: EMA gap ≥ 0.15% of price (not a weak signal) c3 = ind["ema_sep"] >= 0.0015 # Condition 4: Price is within 0.2% of a pivot level pvt_name, pvt_price = PivotCalculator.nearest_pivot(ind["close"], pivots) c4 = pvt_name is not None # Condition 5: Signal candle confirms the direction c5 = ind["is_bullish"] if direction == "long" else ind["is_bearish"] # Condition 6: Volume ≥ 1.5x the 20-bar average c6 = ind["vol_ratio"] >= 1.5 # ALL must pass — one failure = skip this candle if all([c1, c2, c2b, c3, c4, c5, c6]): # → Signal valid! Why all 6 conditions and not just 3?

Each condition filters a different type of bad trade. ADX removes sideways markets. EMA orientation prevents counter-trend trades. EMA separation removes weak crossovers. Pivot proximity gives a specific entry price. Candle direction confirms momentum. Volume confirms institutional participation. Remove any one and you get significantly more losing trades. The 2021–2024 backtest shows the system went from 39% win rate (no filters) to 69% (all 6 active).

Day 3 · Risk Mathematics
Section 7 — SLTPCalc (Stop-Loss & Position Sizing)
The most dangerous number in any trade is the quantity. Get this wrong and a single bad trade can wipe out weeks of gains. Position sizing is not a nice-to-have — it is the core of risk management. This section enforces a strict 3-method SL hierarchy and calculates quantity from a fixed capital risk percentage.
Capital at risk = ₹5,00,000 × 1% = ₹5,000
SL per share = |₹1,200 − ₹1,185| = ₹15
Quantity = ₹5,000 ÷ ₹15 = 333 shares
This means if your SL is hit, you lose exactly ₹5,000 — no more, no matter what price does.
The 3-Method SL Hierarchy
  • 1Pivot-based SL (primary): Place SL 0.4% beyond the pivot level. This is the most logical placement — the pivot is where price should hold if your analysis is correct. If price breaks through the pivot, your thesis is wrong.
  • 2ATR-based SL (alternate): Place SL at 0.5 × ATR from entry. Used when the pivot-based SL would be too far from entry. Ensures the stop is within a meaningful range for the stock's normal volatility.
  • !Hard cap at 0.8%: If either method above results in a SL that is more than 0.8% away from entry price — skip the trade entirely. Do not make exceptions. A wide SL is telling you the setup is risky.

The SL limit vs trigger price — why two prices?

Firstock SL orders need both a trigger price (when the SL order activates) and a limit price (the worst price you'll accept for the fill). Set the limit price 0.3% below the trigger (for longs) to ensure the order actually fills when triggered. If you set both at the same price and the market gaps through your SL, the order may not fill.

Day 4 · Trade Management
Section 9 — PositionMonitor
Entering a trade is only half the job. How you manage it after entry determines your actual P&L. The PositionMonitor checks all exit conditions on every 15-minute candle for every open position. The exit hierarchy is strict — ADX drop is checked first, then the time limit, then SL, then targets.
Exit Priority Order (most urgent first)
  • 1ADX drops below 20 mid-trade → Exit 100% immediately at market price. The trend condition that justified your entry no longer exists. Hold and you will lose the T1 gain you've built.
  • 23:15 PM hard close → Exit 100% at market regardless of position. Never hold intraday positions overnight. Overnight news can gap against you and your SL won't protect you.
  • 3Stop-loss hit → Exit 100%. The analysis was wrong. Accept the pre-calculated loss and move on.
  • 4Target 1 hit → Exit 50% of the position. Immediately move the remaining SL to the entry price (breakeven). Now the worst case on the remaining 50% is a scratch — you have locked in the T1 profit.
  • 5Target 2 hit → Exit remaining 50%. Full trade closed. Log it and move on.

Day 4 · Automated Discipline
Section 10 — DailyRiskGuard
This section is your automated willpower. Every experienced trader knows that the most dangerous moment is after a losing trade — the urge to "make it back" leads to revenge trading and bigger losses. The DailyRiskGuard makes those rules automatic and unoverridable in code.
What DailyRiskGuard enforces automatically

Daily loss limit (2%): If your P&L for the day drops to -₹10,000 (on ₹5L capital), the system halts all trading and sends a Telegram alert. No more trades today.

Max trades per day (3): After your 3rd trade closes, the system halts. Quality always beats quantity.

Why this saves money: On 2 consecutive loss days, discretionary traders often increase position size trying to recover — and turn a manageable drawdown into an account-threatening one. The algo's DailyRiskGuard prevents this by design.

When you set up Firstock 2FA, instead of scanning the QR code, click "Enter setup key manually." The key shown (a string like JBSWY3DPEHPK3PXP) is your base32 TOTP secret. Write it down and store it as FIRSTOCK_TOTP_SECRET environment variable. This is what pyotp uses to generate codes.

Broker Setup

Firstock API Setup — Complete Guide

One-time setup. Takes 30–60 minutes. Free at every step.

  1. Open a Firstock account. Go to firstock.in → Open Account. Complete Aadhaar-based KYC. No charges: ₹0 account opening, ₹0 AMC forever.
  2. Enable API access. After account activation, log in → Profile → API & Developer → Click "Generate API Key." Note your API Key and Vendor Code — you need both.
  3. Set up TOTP manually. When adding Firstock to Google Authenticator, choose "Enter a setup key manually" instead of scanning the QR. Note the base32 key shown (e.g. JBSWY3DPEHPK3PXP). This is your FIRSTOCK_TOTP_SECRET.
  4. Find your instrument tokens. Each stock has a unique token for the API. After logging in via the SDK, run: fs.firstock_searchScrip(userId="ID", stext="RELIANCE") to find any token. Common tokens: RELIANCE=738561, HDFCBANK=341249, INFY=408065.
  5. Set all environment variables. Add to your ~/.bashrc or ~/.zshrc. Run source ~/.bashrc after adding.
  6. Test the login. Run python nse_algo_firstock.py --once. Watch the logs. If you see "Firstock login successful" — you are ready for paper trading.
  7. Paper trade for 30+ days. Run daily at 8:28 AM. Review trades_journal_firstock.csv every evening. Do not skip a day. Do not go live until you have 40+ paper trades with 55%+ win rate.

Learning Path

6-Month Builder's Roadmap

The order of operations for going from zero to live algo — tested by experience, not theory.

Month 1 — Build & Understand
Goal: Write the complete system. Understand every line. Run the backtest. Watch the output. Ask "why did it trade here and not there?" until the logic is completely clear. No live connection yet.
Month 2 — Paper Trading
Goal: Connect to Firstock, run daily paper trades. After each session, review the journal CSV and the log file. Check every signal: did all 6 conditions pass? If the algo skipped a setup, find it in the log and understand why.
Month 3 — Diagnose
Goal: You have 40+ paper trades. Calculate your win rate. Find your weakest exit — is it ADX drops? SL hits? Use the journal to identify patterns. This month is about learning what the algo reveals about the market.
Month 4 — Small Real Money
Goal: Deploy 10–20% of capital with the algo in live mode. Brokerage and slippage are now real. The goal is not profit — it is proving you can operate the system under live conditions without panic.
Month 5–6 — Scale Up
Goal: If Month 4 was profitable, scale to 50% of capital. Two consecutive profitable months with drawdown under 5% is the gate to full capital deployment. Do not rush this step.
Month 7+ — Full Operation
Goal: Full capital deployment. Cron job automates daily startup. Telegram alerts are your eyes. You review the journal on Sunday evenings. Consider adding WebSocket real-time feeds for lower latency.
The most common beginner mistake — and how to avoid it

Skipping paper trading. It feels pointless because "it's not real money." But paper trading reveals how the system actually behaves in live market conditions — flash crashes, illiquid moments, gap opens, sudden ADX collapses — none of which appear in backtest CSV data. Paper trading is not optional. It is the system's final acceptance test.
